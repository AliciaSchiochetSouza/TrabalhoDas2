# ADR-001: Estratégia de Ingestão de Dados — Azure Functions (Serverless)

**Status:** Aceito  |  **Data:** 2026-04-02  |  **Autores:** Equipe DAS II — VendaMais

## Contexto

A VendaMais Distribuidora Ltda. precisa automatizar a extração diária de dados transacionais do seu ERP proprietário (módulos de Vendas, Estoque, Financeiro e Logística) para alimentar a Plataforma de Inteligência Operacional. Atualmente, os relatórios levam até 2 dias úteis para serem compilados manualmente e a diretoria toma decisões com dados de até 30 dias de defasagem.

O pipeline de ingestão precisa atender aos seguintes requisitos:

- **Volume**: ~3.500 pedidos/mês (~117 pedidos/dia) distribuídos em 4 módulos de dados
- **Frequência**: Extração diária (batch), com defasagem máxima de 24 horas
- **Janela de execução**: Processo executado em horário de baixo uso (madrugada), duração estimada de 5–15 minutos
- **Destino**: Arquivos JSON depositados no Azure Blob Storage (camada raw)
- **Custo**: Orçamento limitado — a VendaMais é empresa de médio porte e espera custos operacionais previsíveis
- **Manutenção**: Equipe de TI enxuta, precisa de solução com baixa sobrecarga operacional

O volume de dados é relativamente baixo e a execução é pontual (uma vez por dia), o que torna desnecessária uma infraestrutura permanentemente ativa.

## Decisão

Adotar **Azure Functions (Serverless)** com runtime **Python 3.11** e trigger do tipo **Timer** para implementar a camada de ingestão de dados.

A função será configurada com uma expressão CRON (`0 0 2 * * *`) para executar diariamente às 02:00 UTC. Cada execução conecta na API/banco do ERP, extrai os registros do dia dos 4 módulos (vendas, estoque, financeiro, logística) e deposita os dados como arquivos JSON no Azure Blob Storage.

O plano de hospedagem será o **Consumption Plan** (pay-per-execution), que cobra apenas pelo tempo efetivo de execução e número de invocações.

## Consequências

(+) **Custo extremamente baixo**: No Consumption Plan, o custo é proporcional ao uso real. Com ~30 execuções/mês de ~10 minutos cada, o custo estimado é inferior a R$ 5 por mês — essencialmente gratuito no free tier (1 milhão de execuções/mês gratuitas)

(+) **Zero gestão de infraestrutura**: Não há servidores para provisionar, atualizar, monitorar ou escalar. O Azure gerencia automaticamente a infraestrutura, liberando a equipe de TI de tarefas de manutenção

(+) **Escalabilidade automática**: Se no futuro a VendaMais precisar aumentar a frequência de extração (ex: de diária para horária), o modelo serverless escala automaticamente sem alterações de infraestrutura

(+) **Integração nativa com Azure**: Azure Functions possui bindings nativos para Blob Storage, Timer, e Application Insights, simplificando a implementação e o monitoramento

(+) **Deploy simplificado**: Deploy via Azure CLI, VS Code ou GitHub Actions. Ciclo de desenvolvimento e entrega rápido

(-) **Cold start**: Após períodos de inatividade, a primeira execução pode ter latência adicional de 1–3 segundos (cold start). Para um processo batch noturno, isso é irrelevante

(-) **Limite de execução**: O Consumption Plan tem timeout máximo de 10 minutos por padrão (configurável até 230 min). Para o volume atual da VendaMais, não é um problema, mas pode se tornar relevante com crescimento significativo

(-) **Dependência de plataforma (vendor lock-in)**: A solução fica acoplada ao ecossistema Azure. Migrar para outro provedor de nuvem exigiria reescrita parcial do código de triggers e bindings

## Alternativas Rejeitadas

### Alternativa 1: Máquina Virtual Dedicada (Azure VM)

Provisionar uma VM Linux no Azure para hospedar scripts Python agendados via `cron` do sistema operacional.

**Rejeitada porque:**
- **Custo desproporcional**: Uma VM B1s (a mais barata) custa ~R$ 50–80/mês rodando 24/7, mesmo que o script execute apenas 10 minutos por dia. Relação custo-benefício muito inferior ao serverless
- **Sobrecarga operacional**: Exige manutenção contínua — atualizações de SO, patches de segurança, monitoramento de disco e memória, configuração de firewall
- **Subutilização de recursos**: A VM ficaria ociosa ~99,3% do tempo (10 min de uso em 1.440 min/dia)
- **Escalabilidade manual**: Qualquer mudança de capacidade exige redimensionamento manual da VM

### Alternativa 2: Azure Container Instances (ACI)

Executar a ingestão em um container Docker efêmero, acionado por um agendador externo (Azure Logic Apps ou Azure Scheduler).

**Rejeitada porque:**
- **Complexidade adicional**: Requer criação e manutenção de imagem Docker (Dockerfile, registry) e um orquestrador externo para agendamento, adicionando camadas de complexidade desnecessárias
- **Custo maior que serverless**: Embora mais barato que VM, o ACI cobra por segundo de execução com valores superiores ao Consumption Plan do Azure Functions para cargas leves
- **Sem bindings nativos**: Não possui integração nativa com Blob Storage e Timer como o Azure Functions, exigindo mais código boilerplate para upload e agendamento
- **Overhead para equipe acadêmica**: A equipe precisaria dominar Docker, registries e orquestração — conhecimentos que adicionam complexidade sem benefício para o cenário atual

### Alternativa 3: Azure Data Factory (ADF)

Utilizar o Azure Data Factory com pipelines visuais (low-code) para orquestrar a extração de dados do ERP para o Blob Storage.

**Rejeitada porque:**
- **Custo elevado para o volume**: O ADF cobra por execução de atividade e por Data Integration Unit (DIU). Para o volume baixo da VendaMais (~117 registros/dia), o custo-benefício é desfavorável comparado ao Azure Functions
- **Menor flexibilidade**: O ADF é otimizado para movimentação de dados entre fontes conhecidas. Integrações com APIs customizadas do ERP proprietário podem exigir workarounds complexos
- **Curva de aprendizado específica**: Apesar de ser low-code, o ADF possui conceitos próprios (pipelines, datasets, linked services, integration runtimes) que demandam aprendizado adicional
- **Overhead de licenciamento**: Requer provisionamento de um Integration Runtime e potencialmente de um Self-Hosted IR para acessar o ERP on-premises
