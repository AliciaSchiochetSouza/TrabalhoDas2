# ADR-002: Estratégia de Armazenamento Analítico — Azure SQL Database

**Status:** Aceito  |  **Data:** 2026-04-02  |  **Autores:** Equipe DAS II — VendaMais

## Contexto

Após a extração e transformação dos dados do ERP proprietário da VendaMais (ver [ADR-001](ADR-001.md)), os dados tratados precisam ser persistidos em um banco de dados analítico que sirva como **fonte única de verdade** (single source of truth) para os dashboards do Power BI.

O banco de dados de destino precisa atender aos seguintes requisitos:

- **Modelo de dados**: Estrutura relacional com tabelas de dimensões e fatos (star schema) para atender aos 4 módulos (Vendas, Estoque, Financeiro, Logística)
- **Volume**: ~3.500 pedidos/mês com histórico desde jan/2024. Estimativa de ~200 mil registros na carga inicial, crescendo ~40 mil registros/mês
- **Consultas**: Queries analíticas com agregações, joins e filtros por período, região, representante — padrão típico de BI
- **Conexão com Power BI**: Precisa suportar conexão nativa com Power BI Service (DirectQuery ou Import Mode)
- **Rastreabilidade**: Capacidade de auditar quando dados foram inseridos/atualizados, manter histórico de cargas
- **Custo**: Orçamento limitado, custos previsíveis e proporcionais ao uso
- **Conhecimento da equipe**: A equipe possui familiaridade com SQL relacional (disciplinas anteriores de banco de dados)

O Azure Blob Storage já é utilizado como camada de armazenamento intermediário (raw e curated), mas não substitui um banco relacional para consultas analíticas complexas.

## Decisão

Adotar o **Azure SQL Database** no tier **Standard S1 (20 DTUs)** como banco de dados analítico principal da plataforma.

O banco será organizado em schemas separados por módulo (`vendas`, `estoque`, `financeiro`, `logistica`) e um schema `dw` para tabelas dimensionais e fatos do modelo star schema. A conexão com o Power BI será via **DirectQuery** (ODBC/TDS na porta 1433) para os dashboards que exigem dados atualizados, e **Import Mode** com scheduled refresh para relatórios históricos.

A segurança será implementada com Azure AD Authentication, Transparent Data Encryption (TDE) habilitado por padrão, e firewall rules restritivas por IP.

## Consequências

(+) **Modelo relacional familiar**: SQL é amplamente conhecido pela equipe (disciplinas de BD I e BD II). Não há curva de aprendizado adicional para modelagem e consultas

(+) **Integração nativa perfeita com Power BI**: Azure SQL Database é a fonte de dados mais bem suportada pelo Power BI, com suporte completo a DirectQuery, Import Mode, e scheduled refresh sem configuração adicional

(+) **Rastreabilidade completa**: Com colunas de auditoria (`created_at`, `updated_at`, `load_batch_id`) e o histórico no Blob Storage (camada curated), é possível rastrear a origem e o momento de cada dado inserido no banco

(+) **Custo previsível e acessível**: O tier Standard S1 (20 DTUs) custa ~R$ 75–90/mês e é suficiente para o volume da VendaMais. Sem surpresas de billing por consulta ou throughput

(+) **Segurança robusta nativa**: TDE (criptografia em repouso) habilitado por padrão, Azure AD Authentication, firewall por IP, e auditoria de acesso integrada — sem custo adicional

(+) **Backup automático**: Azure SQL inclui backup automático com retenção de 7 dias (Point-in-Time Restore) sem configuração adicional

(-) **Escalabilidade vertical limitada**: O tier Standard S1 suporta o volume atual, mas crescimento significativo (ex: 10x o volume) pode exigir upgrade para tier Premium com aumento de custo

(-) **Vendor lock-in**: A solução fica acoplada ao Azure SQL. Migrar para outro provedor exigiria adaptação de connection strings, autenticação e configurações específicas. O SQL padrão (T-SQL) é parcialmente portável

(-) **Não otimizado para big data**: Para cenários futuros com volumes massivos (milhões de registros diários), o Azure SQL pode não ser a melhor escolha — soluções como Synapse Analytics seriam mais adequadas

## Alternativas Rejeitadas

### Alternativa 1: Azure Cosmos DB (NoSQL)

Utilizar o Azure Cosmos DB como banco de dados principal, armazenando dados em formato JSON/documento com API SQL.

**Rejeitada porque:**
- **Modelo de dados inadequado**: Os dados da VendaMais são inerentemente relacionais (pedidos → itens → clientes → representantes). Um banco NoSQL exigiria desnormalização e duplicação de dados, dificultando queries analíticas com joins complexos
- **Custo imprevisível**: O Cosmos DB cobra por RU/s (Request Units por segundo). Para queries analíticas com agregações e joins, o consumo de RUs pode ser alto e imprevisível, gerando surpresas no billing
- **Suporte limitado ao Power BI**: A integração Cosmos DB + Power BI exige o conector Cosmos DB, que tem limitações de performance com DirectQuery e não suporta todas as funcionalidades do Power BI
- **Curva de aprendizado**: A equipe precisaria aprender modelagem NoSQL (partitioning, denormalization), o que adiciona complexidade sem benefício para o cenário da VendaMais
- **Rastreabilidade complexa**: Auditoria e versionamento de documentos em Cosmos DB são mais complexos que colunas de auditoria em tabelas SQL

### Alternativa 2: Azure Data Lake Storage Gen2 + Azure Synapse Analytics

Armazenar dados em formato Parquet no Data Lake e usar o Synapse Analytics (serverless SQL pool) para consultas analíticas.

**Rejeitada porque:**
- **Complexidade excessiva para o volume**: A combinação Data Lake + Synapse é projetada para cenários de big data (terabytes/petabytes). Para o volume da VendaMais (~40 mil registros/mês), é uma solução superdimensionada
- **Custo mais elevado**: O Synapse serverless cobra por volume de dados processados em cada query. Mesmo com uso moderado, o custo mensal tende a ser superior ao Azure SQL Standard S1
- **Curva de aprendizado íngreme**: Exige conhecimento de Data Lake, formatos colunares (Parquet/Delta), Synapse Studio, serverless pools, e possivelmente Spark — muito além do necessário para a equipe acadêmica
- **Latência de consulta**: Queries serverless no Synapse podem ter latência de segundos para iniciar (cold start), impactando a experiência de DirectQuery no Power BI
- **Manutenção operacional maior**: Gerenciar partições, compactação de arquivos, e otimização de queries em Data Lake exige expertise específica

### Alternativa 3: PostgreSQL em Azure VM (IaaS)

Instalar e gerenciar um PostgreSQL em uma VM Linux no Azure, controlando totalmente o banco de dados.

**Rejeitada porque:**
- **Sobrecarga operacional significativa**: Exige instalação, configuração, tuning, backup, atualizações de segurança e monitoramento manuais do PostgreSQL e do SO da VM
- **Responsabilidade de backup e DR**: Backup, replicação e disaster recovery precisam ser configurados manualmente, enquanto o Azure SQL inclui tudo nativamente
- **Custo oculto**: Além do custo da VM (~R$ 50–80/mês para B1s), há custo de disco gerenciado e tempo da equipe de TI para manutenção — o TCO (Total Cost of Ownership) tende a ser maior que o Azure SQL PaaS
- **Segurança sob responsabilidade da equipe**: TDE, firewall, auditoria e autenticação precisam ser configurados manualmente, aumentando o risco de falhas de segurança
- **Integração com Power BI**: Requer instalação de On-premises Data Gateway para conectar Power BI Service a um PostgreSQL em VM, adicionando complexidade e ponto de falha
